MQ84 - Topic 6: Problem Determination

Connecting to the queue manager
The request to start the queue manager is issued.
If an error message is returned, indicating that the queue manager is unavailable, look for errors associated with the configuration files.
Errors in accessing the MQSeries configuration file or a misspelt queue manager name can prevent a queue manager from being found. It is important to remember that all MQSeries object names are case sensitive.


Client connection to queue manager
The client application issues a connect call and receives the reason code of ‘queue manager unavailable’.
If the Queue manager was reachable, but unable to accept the connection request, then information will be written by the queue manager to its current information log.
If the failure occurs at the client interface, information will be recorded in error logs on the client machine.

Click both of the information logs for further information, then click Next to continue.


Queue manager connections
If channels between queue managers fail, information will be recorded by each of the queue managers in their current information log.
This information will include the date and time of the event, the information message number and text that describes the event.

Also potential problem resolution activity and prior message relationship is described.

Roll the mouse over either of the information logs to see further information relating to the information logs.


Application errors
Message channel agents record information in the queue manager information logs.

They record when the channel is started and stopped, why the channel was stopped and if retry-able errors have occurred.

The information log messages are text messages, intended to provide for problem tracing and reason analysis.

All messages are recorded with the current date and time at Greenwich and not the local time.

If the channel agents have to put application messages on the queue manager dead letter queue, they will also record in the queue manager’s current information log when and why this was required.


Client application
In the event of the abnormal termination of a client application, an unformatted report will be generated and written to the error directory, provided for by the MQSeries client interface. This report will include the date and time of the error, a snap of the associated memory and problem state information. This information must be interpreted and formatted by a supplied utility program. The output of this utility is provided to assist in the detection and correction of the event. The production of a report does not automatically indicate maintenance to product code is required.


Server connected application
In the event of the abnormal termination of an application connected to the queue manager, a formatted report will be generated by components of the MQSeries product, and stored in an errors directory . The reports will include the date and time of the error, a symptom string, an image of the associated memory locations and problem state information to assist in the detection and correction of the event.

Roll the mouse over the highlighted areas for further information regarding the names and locations and example contents of the event information logs. Then click Next to continue.


MQSeries trace facility
The MQSeries provided trace function allows for detailed execution information to be recorded in a sequential manner. This information is intended to assist in analyzing performance and state problems. IBM service support personnel may ask for a problem to be recreated with trace enabled, as it provides for indepth processing path analysis. The files produced by the trace can be very large, therefore it is important to qualify the trace capture mechanism by providing selection criteria, for example, to trace by time and only record a specific component.

Further information regarding tracing is available in the MQSeries system administration publications.
