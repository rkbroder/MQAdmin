Access Control
Users or groups of users have their required level of authority over MQSeries objects granted or revoked by the MQSeries administrator. The correct level of authority is also required before commands can be issued by a user. The administrator will use the displayed commands to manage these object authority rules.

Click the commands for examples of how they are used.


The message context
When a business process requires that a message be processed by successive applications a security issue arises.

The issue is, under whose authority should all the processing be performed. Should it be the individual authority of each application or a common authority inherited from the originator. MQSeries allows for applications to make this decision at the time of issuing the MQI calls. Also an area in the header of each message is provided to store information related to message ownership.

Click Next to see more details about this ownership information.


The message context
Message ownership information is stored in the context fields of the message header . The message context consists of eight fields divided into two categories. They are the identity context and the origin context.

The identity context identifies the authorized userid at the time of the message being put on the queue. The origin context describes the application that put the message on the queue. A user with the appropriate authority can influence the queue manager in the setting of the context fields.

Alternate security
An application may be required to verify that the originator of a message has the authority to request the services of the next queue. In which case it must use a special option on the mq open call. This option identifies to the queue manager that an alternate userid is to be verified as the requester. While the actual requester is checked for the authority to request this option.


Security and distributed queuing
When a message is sent to a remote queue a number of security checks are possible. Firstly the sending application must have authority to put to the remote object, even if that object is not physically present on the originating system.

Secondly the sending message channel agent must have the authority to get and move the message over the network.

And finally the receiving message channel agent must have the required authority to deliver the message to the requested queue.

Click the transmission queue and the receiving message channel agent for move details regarding their security requirements.
